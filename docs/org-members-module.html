What this branch ships

VineSight already had farmer invitations (onboarding grower clients). This branch adds the parallel, distinct concept of team-member invitations: bringing fellow staff into a consultant organization. The owner/admins can now invite colleagues by email, assign them a role, and distribute the farmer book across agronomists — all from a new Team area in the consultant workspace.

👥 Membership

Invite by email, choose admin or agronomist, list current members sorted owner→admin→agronomist, and remove members (with farmer reassignment cleanup).

✉️ Invitations

Supabase-mailed invite links, one-pending-per-email, 7-day expiry, revoke, copy-link fallback when email can't be sent.

🧑‍🌾 Assignment

Bulk-assign / unassign farmer clients to a chosen agronomist with search, select-all, and live current-assignment labels.

Architecture & invite lifecycle

An invite row drives a state machine. The admin creates it; Supabase Auth mails a passwordless magic link; the invitee confirms their email server-side at /auth/confirm, lands on the accept page, sets a password, and an atomic RPC writes the membership. The status column moves pending → accepted (or revoked / expired).

Data model

One new table, organization_member_invitations, plus a security definer RPC. Shipped both as an incremental migration and folded into the canonical schema dump. The generated TypeScript Database type was updated by hand for the new table.

Key indexes & the partial-unique guard

-- At most one LIVE invite per email per org (case-insensitive).
create unique index idx_org_member_invitations_one_pending_per_email
  on organization_member_invitations(organization_id, lower(email))
  where status = 'pending';

Re-inviting an email expires the prior pending row first, so the admin gets a fresh link instead of a 409 they can't clear. The token column is indexed by its UNIQUE constraint; plus B-tree indexes on organization_id and (organization_id, status).

The atomic accept RPC (defense-in-depth highlights)
  • Role re-validation — rejects anything but admin/agronomist even though the table CHECK already constrains it.
  • Membership insert is ON CONFLICT DO NOTHING — a stale invite can never demote an existing owner/admin.
  • Name preserved — only sets full_name if the profile's is currently null/empty.
  • Status guard — the final update … where status='pending' means two concurrent acceptances (double-fired effect, two tabs) can't both win.
  • Locked down — revoke all … from public/authenticated; only the service role calls it.

API endpoints

All routes authenticate via the session cookie (never trusting an id in the body), validate input with Zod, and use the service-role admin client for privileged reads/writes. Every mutating route re-checks that the caller is an owner/admin of the target org.

EndpointMethodPurposeGuards
/api/organizations/invite-member POST Create invite + send Supabase email Rate-limit (50/min/IP), admin check, reject existing member, expire prior pending
/api/organizations/invite-member GET Public lookup of invite by token (for accept page) Hides revoked/accepted (404), 410 on expiry
/api/organizations/invite-member/revoke POST Mark a pending invite revoked (409 if no longer pending) Admin of invite's org
/api/organizations/accept-invite POST Bind authenticated user to org via RPC Email match, email_confirmed_at required, idempotent re-accept
/api/organizations/assign POST Bulk (re)assign farmers to an agronomist Admin check + assignee must hold agronomist role; writes via RLS client
/api/organizations/remove-member POST Remove member, unassign their farmers, reset profile Can't remove self or owner
/auth/confirm GET Server-side email-OTP verification (token_hash) Same-origin redirect only (open-redirect safe)
Why inviteUserByEmail + /auth/confirm instead of the PKCE ?code= flow? The token_hash path works cross-device — the invitee can open the email in a different browser than the inviter. It also rides Supabase's existing mailer, so no separate sending domain is needed.

Two acceptance paths

The accept page /signup/member/[token] handles two distinct kinds of visitor, distinguished by whether the auth user carries the org_invite_token metadata that inviteUserByEmail stamps on.

New invitee (passwordless)

Arrives authenticated via the magic link with matching org_invite_token. Sees org name + role badge, sets a password (updateUser), then accept-invite runs. On success the token metadata is cleared so a revisit doesn't re-trigger.

Existing account (shared link)

Already has a VineSight account (Supabase won't re-invite), so the admin shares the raw link. They sign in, and acceptance fires automatically in the background → redirect to /consultant. A failure surfaces a "Couldn't join" retry rather than stranding them.

A third, parallel entry exists in /auth/verify-otp: when an inviteToken query param is present, the post-confirmation effect calls accept-invite before redirecting — covering the OTP signup flow.

Frontend pages

  • Team (/consultant/team) — member list with role badges, invite dialog (form → emailed, or copy-link fallback), pending-invites list with copy/revoke, remove-member confirmation. Admin-gated via access.canViewAllFarmers.
  • Assignments (/consultant/team/assignments) — target-agronomist picker, searchable farmer list with select-all, current-assignment labels, bulk assign/unassign. Agronomists get a read-only "Only admins can assign" notice.
  • Accept (/signup/member/[token]) — the dual-path acceptance screen described above.
  • Nav — consultant layout gains a Team nav item; the old "Team Settings — upcoming" placeholder is removed.

Security model

LayerControl
AuthenticationEvery route resolves the user from the session cookie via supabase.auth.getUser() — body-supplied ids are never trusted.
AuthorizationMutations re-verify owner/admin membership server-side; assign additionally checks the target holds the agronomist role.
RLSInvite table is admin-only for all CRUD via is_org_admin(); no public SELECT (acceptance is service-role). Assignment writes go through the authed client so RLS enforces admin-only.
Email verificationaccept-invite requires email_confirmed_at AND a case-insensitive email match — a leaked link can't write a privileged row for an address the actor doesn't control.
Rate limitinginvite-member capped at 50/min/IP (unauthenticated tier) before any work, since each invite spends SMTP quota and creates an auth user.
Open-redirect/auth/confirm only honours relative same-origin next paths.
AtomicityThe 3-step accept (membership + profile + status) runs in one security definer transaction, with concurrency guards on both the insert and the status update.

Observations & follow-ups

  • Generated type is hand-edited. database.ts gained the table by hand, and accept-invite calls the RPC through an (admin as any).rpc(...) cast with an eslint-disable — the comment says to regenerate types after the migration runs. Worth doing to drop the cast.
  • verify-otp sends a body userId. The OTP path posts { userId, token }, but accept-invite ignores body userId entirely (uses the session) — harmless but dead data worth trimming.
  • Expiry is lazy. Rows flip to expired on read/accept rather than via a cron — fine for correctness (every consumer checks expires_at), but the pending list could briefly show a just-expired invite until next load.
  • Two migrations both define schema (incremental file + canonical dump). Confirm the deploy pipeline applies the incremental migrations and treats supabase-schema.sql as reference only, to avoid double-create on fresh DBs.
  • External dependency not in this diff. The cross-device /auth/confirm flow relies on a custom Supabase invite email template that emits token_hash and links to /auth/confirm?next=…. That template lives in Supabase project config, not the repo — the code asserts it (invite-member route comment) but the diff alone can't verify the email renders the right link.
  • Not independently run/tested here — this doc is a read of the diff, not a verification of runtime behavior. Suggested manual QA: invite a brand-new email, invite an existing-account email (link fallback), revoke, re-invite same email, accept, then bulk-assign + remove.
