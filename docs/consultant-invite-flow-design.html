1 Overview

This feature lets an agricultural consultant invite a farmer to their organization by phone number. The consultant generates a single-use, token-bound signup link and shares it over WhatsApp, SMS, or copy/paste from their own device — there is no SMS gateway. When the farmer signs up through that link, they are automatically bound to the consultant's organization as an active client. A companion change centralizes role-based routing in middleware so org staff and farmers each land in — and stay within — their own module.

2 Context & constraints

VineSight is a B2B platform: the buyers are Nashik grape exporters, FPCs, and agronomy consultants; the farmers are the deployed end-users. A consultant's first job after onboarding is to bring their roster of farmers into the platform — so farmer onboarding has to be fast and frictionless for the consultant, who is doing the inviting.

Why deep links instead of an SMS/WhatsApp API

For the MVP we deliberately avoided an integrated messaging provider. A real SMS/WhatsApp Business integration would require provider accounts, DLT-approved templates, and per-message cost — overhead that doesn't fit an early go-to-market. Instead, /api/invite/create returns a ready-to-share link, and the UI renders Copy / WhatsApp / SMS buttons the consultant taps from their own phone. The link is the entire payload; nothing is sent server-side.

Two frontends, one backend

VineSight runs two frontends on one Supabase backend: this Next.js web app and a separate React Native app that owns the farmer experience. The web app is becoming the org/consultant cockpit; the web farmer module is frozen (kept only as a desktop fallback). The invite flow lives on the web side because that is where consultants work — but it onboards farmers who will primarily use the RN app.

3 System architecture

Authenticated requests flow through a single middleware chokepoint that decides which module a user belongs to, based on whether they have a row in organization_members.

Middleware is the single redirect authority — each role is only ever sent to its own home, so the modules can never ping-pong.

4 Invitation lifecycle

The happy path spans two actors: the consultant (in the web cockpit) creates and shares the link; the farmer opens it, signs up, and is bound to the org.

The dashed handoff (consultant → farmer) is the out-of-band share; everything else is server-mediated.

Invitation status states

pending →accept accepted pending →TTL expired pending →admin revoked

The pending → accepted transition is the single-use gate, and it is performed atomically (see §7). The other terminal states are reached by TTL expiry or manual revocation.

6 Data model

Two tables carry the flow. The new invitation columns are nullable so the pre-existing email-invite path keeps working unchanged.

farmer_invitations

ColumnTypeNotes
tokenvarchar(255)unique · the bearer secret in the link (crypto.randomUUID())
statusvarchar(50)default pending · CHECK in (pending, accepted, expired, revoked)
organization_iduuidFK → organizations
expires_attimestamptzcreate() sets now + 7 days
phonevarchar(20)0004 E.164, used to deliver + backfill profile
farmer_namevarchar(255)0004 optional, for personalization + prefill
invited_byuuid0004 FK → auth.users · the farmer is auto-assigned to this member

organization_clients (the binding target)

  • UNIQUE(organization_id, client_user_id) — a user can't double-link to the same org; backs the accept upsert's ON CONFLICT.
  • UNIQUE (client_user_id) WHERE status = 'active' — one active org per client; surfaces a cross-org conflict as a 23505 the accept route maps to a clean 409.
⚠ The constraints do not stop one token binding two users

Neither unique rule prevents two different fresh accounts from accepting the same token (different client_user_ids never collide). That guarantee lives in the application layer — the atomic claim in §7 — not the schema.

7 The accept endpoint

accept is where the security and concurrency work concentrates. It runs on the service-role admin client (the bind crosses RLS while the farmer is mid-signup), but authorizes by phone-ownership proof — the caller's authenticated session must carry a confirmed phone that matches the invitation — and adds an atomic single-use guarantee that survives forwarded links and browser retries.

Request pipeline

1
Load the invitation
Fetch by token on the service-role client (RLS would otherwise hide the row from this caller).
→ 404 not found
2
Prove phone ownership
validateUserSession — the caller's session user must equal the posted userId, and its confirmed phone must digit-match invite.phone. This is the authorization gate, replacing the old account-creation-time window.
→ 401 no session · 403 session/phone mismatch
3
Status & expiry pre-check
A non-pending status is a fast path — for an already-accepted invite it first checks whether this user is already the org's active client (idempotent retry, below); past expires_at flips the row to expired.
→ 410 expired · 409 spent
4
Concurrent reads
One Promise.all: the farmer's profile (for phone backfill), any existing active client link, the inviting org's is_active flag, and the caller's organization_members row.
5
Guards — before claiming the token
Staff guard (a member of the inviting org can't bind as its own client); one-active-org-per-client; and the org must still be active (create's check is up to 7 days stale).
→ 403 staff · 409 other org · 400 org inactive
6
Atomically claim the single-use token
A conditional UPDATE … WHERE status = 'pending'. Of any concurrent callers, exactly one matches a row and proceeds; the rest match zero and stop. This is what enforces single-use — not the read in step 3.
lost race → idempotent-retry check → success or 409
7
Link, rollback on failure & finish
Upsert into organization_clients (active, assigned to the inviter). If it fails, the claim is released back to pending so the farmer can retry. On success, sync the legacy profiles.consultant_organization_id mirror and backfill the invited phone if the farmer has none.
23505 cross-org → 409 · else rollback + 500

The atomic claim

// Reading status then later writing 'accepted' is a TOCTOU race: two concurrent
// accepts of one forwarded link could both pass the read and both bind. The
// conditional UPDATE is the real gate.
const { data: claimed } = await admin
  .from('farmer_invitations')
  .update({ status: 'accepted' })
  .eq('id', invite.id)
  .eq('status', 'pending')   // ← only one concurrent caller matches a row
  .select('id')
  .maybeSingle()
src/app/api/invite/accept/route.ts — Postgres applies the WHERE status='pending' predicate atomically.

Idempotent retries in both timing windows

A single farmer can legitimately hit accept twice — a lost response, or a double-submit. There are two windows, and a benign retry must succeed (not show a scary "linking failed") in both: 

async function hasActiveClientLink(admin, organizationId, userId) {
  const { data } = await admin.from('organization_clients')
    .select('id').eq('organization_id', organizationId)
    .eq('client_user_id', userId).eq('status', 'active').maybeSingle()
  return Boolean(data)
}

// Window A — sequential retry: the first request already committed 'accepted'.
if (invite.status !== 'pending') {
  if (invite.status === 'accepted' &&
      await hasActiveClientLink(admin, invite.organization_id, userId))
    return ok()           // it's me, already linked → success
  return conflict()
}
// Window B — concurrent: we read 'pending' but lost the atomic claim race.
if (!claimed) {
  if (await hasActiveClientLink(admin, invite.organization_id, userId)) return ok()
  return conflict()
}
One helper, two call sites — sequential retries hit the early guard, concurrent ones hit the claim-loss branch.

8 Key design decisions

🔐 Authorize by phone-ownership, not a creation-time window

The bind runs on the service-role client (it crosses RLS while the farmer is mid-signup). To stop it binding arbitrary users, accept requires the caller's authenticated session to carry a confirmed phone that digit-matches the invitation, and the session user to equal the posted userId. This replaces an earlier "account created < 5 min ago" heuristic — it rejects a forwarded link opened on a different number, still lets an already-registered farmer accept, and survives slow OTP entry.

🚫 Block known numbers at create, backstop at accept

create() refuses to invite a number that already has an account, via the phone_in_use(p_e164) SECURITY DEFINER RPC — it reads auth.users.phone (unreachable from app code over PostgREST) plus profiles.phone. An email-only account keeps no phone anywhere, so accept is the reliable backstop: it refuses to bind a userId that is already an organization_members row of the inviting org — staff can't become a client of their own org.

⚛ Single-use is an application invariant, enforced atomically

The schema can't express "one token → one client" (different users never collide on the unique keys). So single-use is enforced by the conditional UPDATE … WHERE status='pending'. Without it, a forwarded link accepted concurrently by two fresh accounts would bind both.

↻ Idempotent by design, not by accident

Retries and double-submits are normal browser behavior. Rather than hope they don't happen, the endpoint treats "this user is already the org's active client" as success in both the sequential and concurrent windows — the farmer never sees a false error after a successful signup.

🏢 Re-validate the org at bind time

create() checks is_active, but that check is up to 7 days stale by the time the farmer accepts. accept() re-reads it (folded into the existing concurrent read, so no extra latency) — mirroring the sibling /api/organizations/add-client path. A since-deactivated org can't silently acquire new clients.

🧭 Gate on the authoritative table, not a cached column

Middleware decides the module from organization_members — the same source getConsultantAccess() uses — rather than profiles.user_type, a derived cache that can lag membership. Keying off the cache risked bouncing a real member off /consultant before the layout could admit them.

📲 Share via deep links, no provider integration

The link is built server-side and shared by the consultant from their own device. No Twilio/Gupshup/DLT, no per-message cost, no template approvals — the right scope for the MVP, and a clean upgrade path if integrated sending is added later.

9 Module gating

Because farmer routes are scattered across ~16 top-level paths with no shared layout, the gate lives in middleware.ts — the one place every request already passes through for auth. 

const { data: membership } = await supabase
  .from('organization_members')
  .select('user_id')
  .eq('user_id', user.id)
  .limit(1)            // safe for multi-org users
  .maybeSingle()

const isOrgUser = Boolean(membership)
// org member → /consultant ;  everyone else (farmers) → /dashboard
  • Org member on a farmer route → redirected to /consultant; non-member on /consultant → /dashboard.
  • The membership lookup only runs on /auth + the two module root-sets — shared pages (settings, help, legal, homepage) pass through untouched.
  • The lookup is indexed (idx_organization_members_user_id) and RLS self-readable, so it's the same cost as the prior profile read.
  • The consultant layout no longer auto-redirects on access failure (it renders an inline message) — so it can't fight middleware into a loop.
⚠ Farmers have user_type = NULL, never 'farmer'

The handle_new_user() trigger never sets user_type; only the org create/join routes write 'org_member'. The gate fails open to the farmer module, so a missing or odd user_type can never lock anyone out — another reason it reads organization_members instead.

📦 Why 0005 exists instead of editing 0001/0003

Migrations 0001–0003 are already applied to production. Applied migrations are immutable — editing them in place would silently never re-run. So the two deltas that were briefly staged as in-place edits were reverted and re-issued as a new forward migration, keeping a live DB and a fresh install converging to the same end state.

🚦 Deploy gate

The flow hard-depends on 0004's columns — create/accept will 500 without them, and create additionally needs 0006's phone_in_use function. All three — 0004, 0005, and 0006 — must be applied to the target database as part of (or before) the deploy.
