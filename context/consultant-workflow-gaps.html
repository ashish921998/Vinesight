Current Branch Review

Consultant, Agronomist, and Farmer Workflow Gaps

Review of the current PR #131 consultant dashboard implementation, focused on how the web consultant workflow should connect with agronomists and farmer users in the React Native app.

Intended Operating Model

1

Consultant Org

FPC, exporter, or advisory business owns the customer relationship.

2

Team Members

Owners, admins, and agronomists access the web consultant dashboard.

3

Farmers

Farmers use the React Native app and are linked as clients.

4

Advisor Work

Agronomists review triage, create plans, and manage follow-up.

5

Farmer Feedback

Farmers receive plans and acknowledge or ask questions in mobile.

Current Data Model

Table Current Meaning Comment
organizations Consultant organization or operating entity. This is the right anchor for the web consultant business account.
organization_members Internal consultant team membership with role. Should be the source of truth for consultant staff access.
profiles User profile fields and consultant_organization_id. Current code uses this to link farmers to an org. It should not be the primary source of consultant staff membership.
organization_clients Farmer/client relationship and agronomist assignment via assigned_to. This should become the canonical client membership and assignment table.
petiole_triage AI classification, review status, and current farmer acknowledgment fields. Works for triage, but acknowledgment overlaps with plan_acknowledgments.
plan_acknowledgments Farmer reaction per fertilizer plan. Better long-term home for mobile plan feedback than triage-level fields.

Proposed Database Mapping

We do not need a separate consultants table for V1. The clean model is to treat the consultant business as an organization, consultant staff as organization_members, and farmers as clients connected through organization_clients.

Concept Table / Columns Action
Consultant company, FPC, exporter, or advisor business organizations Keep. This is the account/workspace boundary.
Owner, admin, agronomist staff users organization_members.user_id, organization_members.role Keep and standardize. Use roles owner, admin, agronomist.
Farmer mobile app user auth.users, profiles, farms.user_id Keep. Farmer remains a normal authenticated user who owns farms and records.
Farmer belongs to consultant org organization_clients.organization_id, client_user_id Make canonical. Read farmer directory and plan builder from this table, joined to profiles and farms.
Farmer assigned to specific agronomist organization_clients.assigned_to Keep. This is the core scoping field for agronomist access.
Petiole AI triage and review workflow petiole_triage Keep. It is org-scoped and farm-scoped.
Reusable consultant fertilizer templates plan_templates Keep. It is correctly scoped by organization_id.
Farmer reaction to a delivered plan plan_acknowledgments Use this as the canonical table. Do not store long-term plan feedback on petiole_triage.

Important Direction

Avoid using profiles.consultant_organization_id as the primary relationship between a farmer and a consultant organization. It is too limited for assignments, history, status, and future cases where a farmer may move between consultants or be served by multiple programs.

Table Changes Needed

Table Change Reason
organization_members Ensure role constraint allows exactly owner, admin, agronomist. Add or keep an ownership flag only if the rest of the app already depends on it. One consistent staff role model across DB, RLS, TypeScript, and UI.
organization_clients Add/confirm columns: id, organization_id, client_user_id, assigned_to, status, assigned_by, assigned_at, notes, created_at, updated_at. This table must represent the business relationship, assignment, and lifecycle of a farmer under a consultant org.
profiles Stop relying on consultant_organization_id for core reads. Either keep it as a denormalized shortcut or plan a later migration to remove it. A single FK on profile cannot represent assignments, status, invitations, history, or multiple relationships.
plan_acknowledgments Add optional columns if mobile feedback needs them: note, created_by_role, updated_at. Farmers may need to say "questions" with a short note. The consultant dashboard can then show plan-level feedback reliably.
petiole_triage Keep triage review fields, but remove or stop using farmer_acknowledgment for plan reactions. Triage and delivered-plan feedback are different lifecycle events.

Suggested SQL Shape

This is not a final migration, but it shows the direction the schema should move toward. It assumes the DB Foundation work is already merged and should not duplicate constraints or policies that now exist in the base schema. 

-- Canonical farmer/client relationship
ALTER TABLE organization_clients
  ADD COLUMN IF NOT EXISTS assigned_to UUID REFERENCES auth.users(id) ON DELETE SET NULL,
  ADD COLUMN IF NOT EXISTS status TEXT NOT NULL DEFAULT 'active'
    CHECK (status IN ('active', 'inactive', 'pending')),
  ADD COLUMN IF NOT EXISTS updated_at TIMESTAMPTZ NOT NULL DEFAULT now();

-- Already expected from DB Foundation:
-- UNIQUE (organization_id, client_user_id)
-- SELECT/INSERT/UPDATE/DELETE RLS on organization_clients
-- fertilizer_plans.organization_id and organization-scoped RLS

-- Optional only if real invites are in V1
CREATE TABLE organization_invitations (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  organization_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  email TEXT NOT NULL,
  invite_type TEXT NOT NULL CHECK (invite_type IN ('team_member', 'farmer_client')),
  role TEXT CHECK (role IN ('admin', 'agronomist')),
  assigned_to UUID REFERENCES auth.users(id) ON DELETE SET NULL,
  invited_by UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  accepted_by UUID REFERENCES auth.users(id) ON DELETE SET NULL,
  status TEXT NOT NULL DEFAULT 'pending'
    CHECK (status IN ('pending', 'accepted', 'expired', 'revoked')),
  token_hash TEXT NOT NULL,
  expires_at TIMESTAMPTZ NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  accepted_at TIMESTAMPTZ
);

Status Values

Current code and schema use active, inactive, and pending for organization_clients.status. Do not introduce invited, paused, or archived unless we also ship an explicit status migration, generated type update, and UI behavior change.

Authentication and Onboarding Flow

All user types should use the same Supabase Auth system. We should not create separate auth models for consultants, agronomists, and farmers. Their product context comes from relationship tables after sign-in.

User Type How They Sign In Rows Created / Linked Resulting Access
Consultant owner Web signup with Google or email/password. auth.users, profiles, organizations, organization_members(role = owner). Can access consultant dashboard, manage org settings, team, and all farmers.
Admin Web signup/sign-in, then added to consultant org. auth.users, profiles, organization_members(role = admin). Can manage most org operations, farmers, templates, assignments, and team.
Agronomist Web signup/sign-in, then added to consultant org. auth.users, profiles, organization_members(role = agronomist). Can access consultant dashboard and work assigned farmers.
Farmer React Native app signup/sign-in. auth.users, profiles, farmer-owned farms, and organization_clients row when linked to a consultant org. Can use the mobile app, own farm data, receive plans, and acknowledge plans.

V1 Onboarding Recommendation

For pilot testing, onboard consultant owners through web signup, manually add agronomists to organization_members, and link farmers through organization_clients. Add real invitation flows only when we need self-serve onboarding.

Out Of Scope For V1

Self-Serve Invites

Real email invite acceptance can wait unless pilot onboarding depends on it. Do not keep a fake invite success state.

Billing

Billing can be modeled later at the organization level. It should not block the consultant dashboard data model.

Multi-Org Switching

For V1, assume one consultant organization per staff user or explicitly choose the first membership. Add an org switcher later.

Audit Log

A consultant activity log is useful for compliance, but not required to validate the core consultant-farmer workflow.

Notification Delivery

Real email/SMS alerts require server-side preferences and delivery jobs. LocalStorage settings are only UI state.

Viewer Role

Viewer should wait until DB constraints, RLS, UI permissions, and product need are all aligned.

Main Gaps

This section tracks the current post-DB-foundation state. It separates issues already solved in the base schema from issues still present in the consultant dashboard branch.

Priority Gap Why It Matters Recommended Change
High Farmer directory still reads profiles.consultant_organization_id. /consultant/farmers and the farmer detail page still use the legacy profile FK as the primary relationship. That bypasses organization_clients.assigned_to and breaks agronomist-scoped visibility. PR 2 must move all farmer directory reads to organization_clients, join profiles only for display fields, and apply owner/admin versus assigned agronomist filtering in one shared query helper.
High /consultant role access must stay aligned with DB roles. Owners and admins can manage the organization and should not be blocked from consultant pages. Authorize owner, admin, and agronomist. Gate capabilities inside pages based on role.
High Role definitions are inconsistent. DB allows owner, admin, agronomist, while Settings UI still includes viewer. The database constraint does not support that value, so the invite dialog can fail at the DB layer if it is used. For V1, remove viewer from UI and types. Use one shared role union: owner | admin | agronomist.
High Invite flow is fake. Settings shows invitation success, but it only sleeps and clears the form. No user, member, invite, or email is created. Either hide invites for V1 or implement a real invite table/API that creates pending organization membership.
High Farmer-client relationship is still split between two models. Some code reads farmers through profiles.consultant_organization_id, while RLS and assignment use organization_clients. Make organization_clients canonical. Keep profiles.consultant_organization_id only as a denormalized convenience if needed.
High Current branch SQL depends on DB Foundation tables and policies. The branch migration references fertilizer_plans.organization_id and organization_clients behavior that must come from the merged base schema. Without that prerequisite, plan RLS and upserts can break. Treat DB Foundation as a hard prerequisite. Do not duplicate the schema in later PRs, but call out the dependency clearly in migration notes and PR descriptions.
Medium Acknowledgment is implemented in two places. petiole_triage has farmer_acknowledgment, and plan_acknowledgments has reaction rows. Mobile plan feedback should be plan-level. Use plan_acknowledgments for farmer app reactions. Remove or reserve triage acknowledgment for triage-specific status only.
Medium Triage farmer acknowledgment RLS is too broad. The current policy lets farm owners update triage rows through a self-join pattern and does not restrict updates to acknowledgment-only fields. In the triage PR, replace this with a narrow RPC or stricter RLS that only permits farmer-owned acknowledgment fields to change.
Medium get_farm_clusters does not apply agronomist assignment scope. The function checks organization membership, so an agronomist can see org-wide cluster data rather than only farmers assigned through organization_clients.assigned_to. Restrict cluster reads by role: owner/admin can see all org clients; agronomists only see assigned clients. Alternatively keep clusters out of PR 2 and fix them in a later settings/polish PR.
Medium React Native contract is not specified enough. The doc says farmer plan feedback should align with plan_acknowledgments, but it does not identify current RN calls, API routes, or payload changes. Add a contract discovery step before Farmer Feedback: list current RN endpoints, plan read path, acknowledgment payload, offline behavior, and required web API/RLS changes.
Medium organization_clients.status values were inconsistent. Earlier SQL examples used invited, paused, and archived, while current code/schema use active, inactive, and pending. Keep the current values for now. Only change them in a dedicated migration with generated types, UI mapping, seed cleanup, and tests.
Medium Notification preferences are stored in localStorage. They do not travel across devices, users, or browsers, and cannot drive server-side alerts. Add organization_notification_preferences or user_notification_preferences after V1 if alerts are real.
Medium Organization selection assumes one membership. Code frequently uses .limit(1).maybeSingle(). A consultant may later belong to multiple organizations. For V1, explicitly enforce one org per consultant user or add an org switcher.
Low Copy says "agronomist consultants only". It conflates role and product area, making owner/admin access confusing. Use "consultant team members" for area-level copy.

Already Covered by DB Foundation

These are not open gaps in the current schema if the merged DB Foundation work is present. Keep them as prerequisites, but do not reimplement them in the smaller feature PRs.

Area Current Position What Later PRs Should Do
fertilizer_plans.organization_id The base schema includes organization scoping and organization-member RLS for fertilizer plans and plan items. Later plan work should rely on the existing column and policies, then add only feature-specific reads, writes, and tests.

Recommended V1 Changes

1. Normalize Roles

Create one shared role definition used by UI, services, database types, and RLS: owner, admin, agronomist.

  • Owner/admin can access all organization consultant pages.
  • Agronomist can access assigned operational work.
  • Remove viewer from V1 until implemented end to end.

2. Add Membership Helper

Add a single helper such as getCurrentOrganizationMembership() and use it across consultant pages.

  • Returns organization_id, role, and permissions.
  • Avoids each page reimplementing auth and role lookup.
  • Makes future org switching easier.

3. Canonicalize Clients

Move farmer/client membership reads to organization_clients and join profiles only for display.

  • Use assigned_to for agronomist scoping.
  • Owner/admin see all org clients.
  • Agronomist sees assigned clients.

V1 Position

Do not add more abstractions than needed. Keep one consultant organization, one team membership table, one canonical farmer-client table, and one assignment field. That is enough to support the consultant, agronomist, and React Native farmer workflow.

Testing Strategy

We cannot prove the feature is entirely bug-free, but we can make the important failure modes repeatable and hard to miss by testing the role matrix at the database, service, page, and end-to-end levels.

Seeded Persona Matrix

Persona Purpose Expected Access
Consultant owner Organization owner and admin-equivalent operator. All org farmers, all triage, templates, settings.
Consultant admin Non-owner manager. All org farmers, team operations except owner-only actions if any.
Agronomist A Assigned operator for Farmer A. Only Farmer A and Farmer A's farms/triage/plans.
Agronomist B Assigned operator for Farmer B. Only Farmer B and Farmer B's farms/triage/plans.
Farmer A React Native app user assigned to Agronomist A. Own farms, own plans, own acknowledgments.
Unrelated user Negative access test. No consultant dashboard, no org farmer data.

Testing Rule

Every PR should include tests for the permission branch it changes. If a page shows different data for owner/admin versus agronomist, both paths need explicit test coverage.
